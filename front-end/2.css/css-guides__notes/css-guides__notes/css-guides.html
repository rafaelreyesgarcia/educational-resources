Flexbox

Basics

    main axis

    defined by flex-direction

    row, row-reverse | main axis is inline direction (left to right or reverse) , (x axis), row.

    column, column-reverse | main axis is block direction (top to bottom or reverse), (y axis), column.

    cross axis

    perpendicular to the main axis.

    • flex-direction set to row, cross axis is column (y axis, block direction).
    • flex-direction set to column, cross axis is row (x axis, inline direction).

    start and end lines

    past bias towards writing modes, makes flexbox

    • flex-direction set to row, start on left, end on right.
    • flex-direction set to row-reverse, start on right, end on left.
    • flex-direction set to column, start on top, end on bottom.
    • flex-direction set to column-reverse, start on bottom, end on top.

    flex container (parent)

    defined by setting the display to flex or inline-flex.

    Default Values

    • items display in row.
    • items start from the start line of the main axis.
    • items do not stretch on the main dimension, but shrink.
    • items stretch to fix size of cross axis.
    • flex-basis is set to auto
    • flex-wrap is set to nowrap

    flex item (children)

    available space

  • left over space in the container after fitting the total items. This applies for both inline and block dimensions.

    • flex-grow

  • property used to distribute available space in proportion.
  • items can grow along the main axis from the basis size.
  • growing will cause items to stretch and take up the available space (or shared available space if other items also grow).
  • a positive integer, as long as it's the same number across items, means that the available space will be shared equally among items.
  • the number is a ratio of how much the items will grow to fill the available positive space.
  • individual flex-grow factors are summed and the positive available space is divided by that sum.

    • flex-shrink

  • property controls how available space is taken away from items.
  • the shrink factor is multiplied by the basis when distributing negative space.
  • distributes negative space in proportion to how much the item is able to shrink.
  • items will stop shrinking at their min-content size.
  • if the container can't fit all items, setting shrink to a positive integer will make the item to become smaller than its basis.
  • higher values will mean that items will shrink faster than items with lower values.
  • positive value will shrink the items in order to not overflow the container.

    • flex-basis

  • size of the item before grow or shrink are triggered.
  • default value is auto. Browser looks at the item's size (width and height).
  • if item doesn't have a size, the content size is used as the basis.
  • auto keyword sets the item size according to its absolute size. If it isn't defined, the browser resolves the size of the content with auto using max-content.
  • using the content keyword, the basis size will go according to the content size, ignoring if width is already set.
  • to completely ignore the size of the item, basis should be set to 0. all the space is available to be shared.

    • flex shorthand

  • 1 1 auto sets both grow and shrink ratio the same. The basis will size the items according to max-content (auto's default choice), then some items will be bigger after distributing the positive available space.
  • 1 1 0 sets the basis size to 0, so all items, regardless of how big their content is, will equally share the available space.
  • starting from a basis of 0, grow and shrink behave more predictably.

    • align-items

  • will align items on the cross axis.

    • align-self

  • overrides align-items to align the item on the cross axis independently from the rest.

    • justify-content

  • aligns items on the main axis.
  • distributes space between or around items.
  • there aren't properties to align individual items on the main axis.
  • setting the value of auto for a margin, will allow the item to take up all of the available space it can. This is called split navigation. Adding margin to items and a negative margin to the container can create gaps in this method.

    • aliign-content

  • sets the distribution of the available space alont the cross axis
  • has no effect on single line containers (set with nowrap)
  • its for containers with multiple lines.

    • gap

  • sets the gap between rows and columns.
  • shorthand for row-gap and column-gap

    • order

  • items are assigned an integer representing their ordinal group.
  • Items with the same integer are grouped and their order in the group depends in the source order.
  • lowest value integer goes first in the group. (ex.1)
  • flex values have a default order of 0.
  • items with an integer greater than 0 display after items without an assigned order value.
  • using a negative integer will ensure that item will always be displayed first.
  • order, just like flow-direction, change the item's display but not the sequential navitagion of them.
  • if you change the visual order, the logical order should also be changed to mirror the behavior.
  • never try to modify the visual order to fix issues with the source. That will cause accessibility issues.
  • usueful situations when visual and logical order are disconnected.
    • dates showing on top of a card but screen reader reading the title first and then the date.

  • flex-grow ratios can be set with decimals (0.25 0.25 0.50) easier to understand that the sum represents all the positive space available to be distributed.
  • auto margins are useful on flex items to absorb space and create gaps between items.
  • There's no way to tell items in one line to line up with the items in the line above, so each flex line acts like an independent flex container.
  • flexbox only deals with space distribution across the main axis.
  • if you want space distribution, row by row, flexbox is good, otherwise, grid.
  • lining up items in the cross axis can be done by controlling the width in percentages, but its bad practice, best practice would be using a wto dimensional grid layout.
  • collapsed items might end up on a different line that they started in a multi-line flex container. Preventing this can be done by wrapping each "fixed" row in a separate flex container.
  • display:none removes the item from the formatting structure. using visibility:hidden keeps the box in the formatting structure and makes the item behave as if it were part of the layout.

    • common flexbox usecases

  • navigation.
  • card layout pushing the footer down. content area set to flex 1 1 0.
  • media objects (image or other element to one side and text to the other side). Let the image determine the sizing for itself, then the content flexes to take the remaining space with flex 1 1 0. preventing the image to grow too large max-width can be applied. Flipping the media object can be done with row-reverse.
  • form controls. Pairing input and button element. flex 1 1 auto for the input allows it to grow while keeping the button the same. A label or icon can be added to the wrapper.

Grid

basics

  • two dimensional layout system.
  • set of intersecting horizontal and vertical lines defining columns and rows.
  • flexbox works with content out and grid works with layout in.
  • track sizes can be fixed (px) or flexible with percentages or set with fractions (fr).
  • placing items can be done with line numbers, names or targeting an area of the grid. This is an explicit layout.
  • implicit layout refers to items not having an explicit position in the grid. Gridplaces these items with an algorithm.
  • alignment can be done to items and for the entire grid.
  • the grid layout specification can add rows and columns if needed. They are part of the implicit layout.
  • more than one item can be placed into a grid cell or area. They can overlap. z-index treats items as layers.

    • grid container

  • set with display:grid or display: inline-grid
  • direct children of the container element (parent) become grid items.
  • grid-template-areas and grid-template-rows(columns) are used on the container
  • the full stop character " . " can leave grid cells empty.
  • each grid area spans multiple grid cells by repeating the name of the grid area.

    • grid tracks

  • space between any two lines on the grid. Defined with the grid-template-row and grid-template-column properties applied on the container. grid-auto-columns and grid-auto-rows specify the size of an implicitly created grid track.
  • auto size results in the content of the tracks determining the size.
  • tracks can be defined with any length unit (absolute, relative) and combine them.
  • tracks can be assigned using the repeat() notation. (how many times to repeat, what value to repeat)

    • the fraction unit (fr)

  • a fraction of the available space in the grid container.

    • minmax( ) function

  • rows and columns can have a minimum size and a maximum size.
  • auto determines the size of the row or column to the biggest content size of an item.

    • grid lines

  • lines are numbered according to the writing mode of the document.
  • lines can be named.
  • indexed from 1.
  • grid-column(row)-start and grid-column(row)-end define the start and end lines of an item.
  • grid-column and grid-row are shorthands for start line / end line.
  • Default is spanning one track. Defining the end line is not necessary if item spans only one track.
  • span keyword specifies how many tracks an area will span.

    • grid cells

  • smallest unit on the grid.
  • children lay out in one cell each of the explicit grid.

    • grid areas

  • when items span one or more cells both by row or column.
  • areas can only be rectangular, not l-shaped.
  • block-start, block-end, inline-start, inline-end
  • counting backwards is possible with a negative integer. The final line is from the explicit grid.

    • gutters

  • also called alleys, are gaps created between grid cells with the column-gap and row-gap or shorthand gap.
  • grid first version prefixed the properties with grid. Old safe versions grid-column-gap grid-row-gap grid-gap
  • space used by gaps is taken into account before assigning fractions to tracks.
  • the gap acts like a thick line separating tracks. Nothing can be placed in them.
  • gaps only appear between tracks, they don't add space to the top, bottom, right or left of the container.
  • gaps act as if the line has gained extra width.

    • nesting grids

  • a grid item can become a grid container.
  • nested grid has no relationship to the parent.
  • working draft of subgrid will allow nested grids to use the track definition of the parent grid.

    • layering items

  • z-index property allows us to control the order in which overlapping items stack.

    • box alignment in grid layout

  • aligning items is done in the block axis and inline axis.
  • align-self and align-items control alignment on the block axis.
  • items with an intrinsic aspect ratio behave as start instead of default stretch.
  • justify-items and justify-self align items on the inline axis. These can't be implemented in flexbox, instead it uses the justify-content property
  • place-items and place-self are shorthands.
  • aligning the grid tracks can be done with align-content for block axis and justify-content for inline axis.
  • auto margin techiques can be applied to items.
  • the logical properties specification allows to use physical properties (top, right, bottom, left) as inline or block.

    • progressive enhancements

  • feature queries (@supports) check if a css property is supported by the browser, if it does, it runs the code, if it doesn't, an old technique like floats is used.

    • subgrid

  • setting the value of subgrid on grid-template-columns(rows) on an item, allows nested grids to have tracks the same size of the parent.
  • gaps are inherited but can be overriden.
  • line names are passed down from the parent, but the subgrid can declare its own line names.
  • line numbering restarts in the subgrid.
  • both rows and columns can be defined as subgrid in the nested grid.
  • there's no implicit grid in subgrids.

    • masonry layout

  • value set for grid-template-columns(rows)
  • layout consists of one grid axis (column), and one masonry axis (rows) where items rise up to fill gaps.
  • align-tracks (block) justify-tracks (inline) masonry-auto-flow are properties to control the masonry axis.

    • useful tips

  • auto-fill keyword can replace the integer value in repeat notation to create as many column or row tracks as will fit.
  • auto-fit replaces integer value in a repeat () function. Size value of a repeat () function can be a minmax( ) function.
  • grid interacts with absolutely positioned elements. To make the grid container a containing block, the position value should be relative. Then the position of a grid item can be absolute.
  • If the grid container doesn't define the relative position, the item will be absolutely positioned to the viewport. The grid item won't participate in grid layout in this case.
  • A grid area can also have a relative position so a nested item can be absolutely positioned to that area.
  • display: contents removes the box an item would normally create and the boxes of the child elements rise up a level. Auto placement rules apply to these child elements. Short term solution for what subgrids are solving.
  • The grid can change with media queries. Start with a mobile grid (single column) and expand the columns as the width of the viewport gets wider.
  • main usecase of grid is to create layouts, but it can also be useful for UI elements.
  • shorthands are useful but if properties aren't or can't be set in the shorthand, they will be reset to initial values.
  • grid-template shorthand sets the explicit grid, and the grid shorthand sets both the explicit and implicit grid.
  • naming grid lines inside square brackets is done in grid-template-rows(columns).
  • naming grid lines multiple names as they might refer to the start or the end of an area.
  • repeat( ) can create multiple grid lines of the same name. addressing different lines is done with an integer.
  • auto-placement usecases can be useful when items don't have a logical order.
